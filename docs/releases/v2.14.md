# v2.14 — feel & spectacle

Agent: **Claude (Opus 4.8), via Hetzner MCP**. Directed by @jonathanwxh-cell.
Date: 2026-07-24. PR [#33](https://github.com/jonathanwxh-cell/gracefell/pull/33).

A directed follow-up to a read-only, gamer's-eye polish review. The review's verdict was that the
combat *feel* is already strong — hit-stop that freezes the simulation but keeps rendering,
perfect-dodge slow-motion, a look-ahead follow-camera, layered reward juice — so this is a
last-10% pass, not a rebuild. Three items shipped here. Two more the review raised (a
hold-to-charge heavy, a phase-three musical lift) were deliberately split into a later "offense"
pass because they need touch-input surgery and audio-bus work that deserve their own QA lane.

## 1. Dynamic combat camera

The follow-camera already leads the player and weights toward the player/boss midpoint, but
`camZoom` was a static viewport-fit — and on a 390px phone that fit is pinned to the **0.55 floor**,
i.e. the most zoomed-*out* the game ever is, on the smallest screen, exactly where reading is
hardest (and where the thumbs cover the corners).

`Game.combatZoomTarget()` now eases the zoom between a calm ceiling (**1.16× the fit**) and the fit
floor by a combat-intensity signal:

```
intensity = clamp(threats / 5 + (phase - 1) * 0.5, 0, 1)   // threats = projectiles + rings + meteors
target    = baseZoom * lerp(1.16, 1.0, intensity)
```

A clean one-on-one tightens for intimacy; the instant area-denial fills the arena — or a later
phase begins — it widens back to the fit so the player never zooms *into* a storm they then can't
clear. During any non-`fight` state the camera eases to the plain fit, so intro/title/terminal
framing is unchanged. The function is exposed the way `menuGeom()` is, so QA reads the intent
numerically instead of eyeballing a screenshot.

## 2. Stagger execution

Breaking Malakar's poise is a real investment (~7–8 openings) that previously paid only 1.4× for
the stagger's duration — a soft reward for the loop's climax. The first **heavy** into a staggered
sovereign is now a riposte: a `30 × 2.6` base which, because `Boss.takeDamage` still applies the
×1.4 staggered multiplier, lands **~109** against a normal staggered heavy's **42**. It fires
exactly once per poise break (`Boss.executeConsumed`, reset in `triggerStagger`), then reverts to
the normal multiplier — a punish *spike*, never a stagger-lock. All juice stays inside the existing
gates: `shake(14, 0.4)` (photosensitivity-gated), a spirit-hued `EXECUTE`, gold flash, `zoomPunch`,
extra hit-stop, and the existing `audio.stagger()` cue. No new Web Audio nodes; the reserved danger
hue is never used. The +5 IRONBOUND capstone never staggers, so it never executes — consistent with
its no-stagger contract.

## 3. Arena deterioration

Phase transitions now call `Game.deepenArena(phase)`, which stamps a ring of scorch cracks into the
**offscreen scorch canvas** — the accumulation surface, never per-frame `drawArena` detail (the
baked-floor invariant) — and raises ambient ember density (`emberDensityMul`, consumed only as a
spawn-rate divisor). `phaseDecay` (1 at phase 2, 2 at phase 3, reset in `resetFight`) makes the
escalation assertable. So "THE SOVEREIGN BURNS" and "GRACE ABANDONS HIM" now *look* like what the
banners say, at zero new render surface, sprite, or particle-budget cost.

## Changed from earlier passes

- **Camera** (was: static viewport-fit). `Game.baseZoom` is now stored and `camZoom` eases toward
  `combatZoomTarget()` while `state === 'fight'`; every non-fight state still eases to the plain fit.
- **Staggered heavy** (was: flat 1.4×, v2.0/v2.10). The *first* heavy per stagger now executes for
  more; every subsequent staggered hit is unchanged.

## QA / verification

Three desktop-lane assertions were added to `qa/verify.cjs` **first** (TDD), each written
defensively so a missing API fails cleanly rather than crashing the harness:

- **Camera intent** — clean melee (threats cleared) target `>` fit × 1.04; eight live projectiles
  target `≤` fit × 1.02.
- **Stagger execution** — force a stagger, position the player, `playerStrike(true)`: `dExec > 84`
  and `dExec > 1.8 × dNormal` and `executeConsumed === true`; a second staggered heavy is normal.
- **Arena decay** — `phaseDecay ≥ 1` after the phase-2 transition and strictly greater after phase 3.

All three failed with pristine engine code (the RED run: `base:null`, `decay:null`,
`dExec:42 == dNormal:42, consumed:false`) and pass after implementation. `npm run lint` and
`npm run build` are clean.

**Environment note.** On the shared dev box (40+ `systemd --user` services) the timing-sensitive
first-gesture audio-init budgets (20/25 ms) blow to 26–101 ms and fluctuate run-to-run. They are a
wall-clock CPU measurement, are unrelated to this pass (no `audio.ts` change), and were already
failing on pristine code in the RED run. The authoritative gate is `.github/workflows/qa.yml` on a
clean `ubuntu-latest` runner, where these budgets pass.
