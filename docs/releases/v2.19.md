# Gracefell v2.19 — battle navigation and score chronicle

Date: 2026-07-24
Implementation: Codex (GPT-5 / Codex Desktop)
Review input: mobile-beginner, veteran-desktop, and audio/accessibility player agents
Status: merged, deployed, and publicly verified

## Outcome

Gracefell now lets a player leave an active battle safely and review dated
victory results from the main menu. The same pass makes existing combat state
clearer without changing the boss, difficulty curve, player timing, score
formula, soundtrack, or performance budget.

## Fixed by Codex

### Battle MENU and return flow

MENU appears after the title in intro, fight, defeat, and victory. It opens a
DOM confirmation that:

- meets the 44 px touch-target floor;
- traps focus and removes the canvas/background controls from the tab order;
- pauses simulation and suspends audio during a live fight;
- clears combat input before resuming;
- states that the current battle will be abandoned;
- returns through `Game.returnToTitle()` without altering earned wins, stored
  scores, settings, or the attempt counter.

The mobile control moves to the second top row, keeping MIX, PAUSE, SOUND, the
player resource bars, and the central playfield readable.

### Title SCORES chronicle

SCORES opens a responsive, scrollable title-screen dialog. It reports total
wins/attempts and retains the latest 20 local victories, newest first:

- grade;
- fight time;
- selected Journey/Measured/Oath path;
- completion date and time;
- damage dealt;
- wounds taken;
- flasks used;
- perfect dodges;
- attempt number.

Empty history has a purposeful first-victory message. Closing by button, Escape,
or backdrop restores focus to SCORES.

### Save schema v6

`Game.SAVE_VERSION` is now 6. `scoreHistory` contains the existing `VictoryScore`
fields plus:

```ts
completedAt: string | null
```

New victories write an ISO timestamp at the same synchronous point where the
score and win are awarded. v5 saves did not know when their last result
happened, so migration preserves that result with `completedAt: null`; the UI
shows “Date unavailable.” No date is fabricated.

`isScoreHistoryEntry()` rejects malformed scores and missing/invalid dates.
Loaded history is capped at 20 even if external data contains more entries.

### Combat feedback and accessibility

- HP, STAM, and FLASKS are labelled on the HUD.
- Light inputs buffered behind an active light animation show `INPUT QUEUED`
  or `NEXT` diamonds. Connected-hit `CHAIN 1/3 → 2/3 → LIGHT FINISHER` scoring
  remains separate and unchanged.
- Narrow non-touch instructions wrap into two lines; desktop bottom help moved
  clear of the viewport edge; title helper text no longer touches the Path plate.
- The semantic companion exposes player health/stamina/flasks, boss
  health/phase/poise/action, combo hits, and queued inputs in a non-live
  definition list.
- The polite live status is reserved for battle state, phase, and telegraph
  changes rather than rapidly changing percentages.
- Keyboard focus is now explicitly regression-tested to expand the companion
  panel to a visible surface with `clip-path: none`.

### Web Audio pause-owner race

Acceptance created a rapid MENU-resume → manual-pause sequence that exposed an
asynchronous state race. An in-flight `AudioContext.resume()` could finish after
the later pause request and make audio run under a frozen game.

`GameAudio.queueContextState()` now serializes context state reconciliation.
After each awaited state change it checks the newest requested owner and
converges again before restarting soundtrack playback. The final owner—
gameplay, MENU, manual pause, or browser interruption—wins.
