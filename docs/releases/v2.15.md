The offense half of the gamer's-eye polish review, deliberately split from v2.14 because both

## Hold-to-charge heavy

The review's finding: offense is one-note (a light 3-chain plus a fixed heavy) against a 1350-HP
boss. The obvious fix — a charged heavy — normally means firing the heavy on *release*, which adds
latency to the basic verb. Instead this charges by **holding through the existing wind-up**:

- A quick tap fires today's heavy **unchanged** — same press timing, same 0.62 s commit, `charge01`
  stays 0, so every existing heavy interaction is byte-for-byte identical.
- If HVY is *still held* when the strike frame arrives (elapsed ≥ 0.42 s), the player **roots in
  place** and `heavyChargeT` builds to `Player.HEAVY_MAX_CHARGE` (0.5 s). Release — or reaching max
  charge — fires a strike scaled by `charge01`: up to **1.75× damage**, +16 range, and a
  `charge*40` **poise bonus** so charged heavies become *the* poise-breaker that sets up the v2.14
  stagger execution.
- The cost is exposure: you stand wound-up and telegraphed — real Souls charged-R2 risk/reward. A
  gold ring tightens and brightens as it charges (reserved danger hue untouched).

Input: keyboard and right-mouse read `input.held.heavy`; touch needed new plumbing.
`Input.touchPoints` now tracks active touches by identifier (start/move/end), and
`Game.heavyInputHeld()` reads `touchLayout()` — the on-screen-controls SSOT — to detect a finger
resting on the HVY fingertip zone (1.6× radius, matching the expanded-target philosophy). The
discrete heavy still fires on press / touch-start exactly as before; charge is purely additive. No
change to `queuedLightAttacks`, roll priority, or one-action touch targeting.

## Phase-3 musical lift

`audio.setPhase()` stored the phase but was **silent** — the adaptive music escalated only by
boss-HP `intensity`. `setPhase` now marks a dirty flag and `updateCombatState()` (still the sole
owner of music transitions) folds a `phaseLift` term (**0 / 0.08 / 0.2** for phases 1 / 2 / 3) into
the **existing** procedural buses: more percussion (`drumsMusic`), a stronger tension pad, and an
open filter on the soundtrack. No new Web Audio nodes, no MP3 change (so no `SOUNDTRACK_VERSION`
bump), and the `master → compressor → peakLimiter` route and bus-gain contracts are untouched. Phase
3 "grace abandons him" now *sounds* like the spectacle v2.14 made it *look* like.

## Changed from earlier passes

- **Heavy** (was: fixed 0.62 s commit). A quick tap is the old heavy exactly; only *holding past the
  strike frame* is new. Damage/range/poise scale with `charge01`, which is 0 for a tap.
- **`audio.setPhase()`** (was: inert). Now drives a sustained phase lift via `updateCombatState`; the
  boss-HP intensity curve is unchanged and adds on top.

## QA / verification

Four assertions were added to `qa/verify.cjs` **first** (TDD), all failing on pristine code and
passing after implementation:

- **Charge on hold** — a real keyboard `KeyK` hold builds `player.heavyChargeT ≥ 0.4`
  (sim-gated via `waitForFunction`, not wall-clock).
- **Charged damage** — with `charge01 = 1`, a heavy deals `> 1.4×` a `charge01 = 0` heavy.
- **Touch held-detection** — a `touchPoint` over the HVY button makes `heavyInputHeld()` true, empty
  makes it false.
- **Phase-3 audio lift** — `audio.debugState().phaseLift ≥ 0.2` after the phase-3 transition.

`npm run lint` and `npm run build` are clean. Real touch-and-hold on a phone is covered by the
detection unit test plus an owner playtest — Playwright's touchscreen has no press-and-hold
primitive. The only remaining local failures are the load-sensitive first-gesture audio-init budgets
(20/25 ms), i.e. shared-box CPU contention; they pass on the clean CI runner and are unrelated to
this pass.
